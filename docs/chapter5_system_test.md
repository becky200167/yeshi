# 第五章 系统测试与结果分析

为验证智慧夜市空间信息可视化平台的稳定性、功能完整性与权限控制有效性，本章从功能测试、接口异常测试以及轻量性能分析三个方面对系统进行验证。测试对象覆盖普通用户、商户用户与管理员三类角色，重点考察以下内容：

1. 系统核心功能是否能够按照设计要求正常运行。
2. 基于角色的访问控制机制是否能够有效限制越权操作。
3. 摊位审核与评价互动等关键业务闭环是否能够完整执行。
4. 系统在当前数据规模下是否具有可接受的响应速度与基本可用性。

本章测试采用“浏览器功能验证 + Flask `test_client` 接口测试 + SQLite 数据校验”相结合的方法进行。为了避免污染原始业务数据，自动化测试在复制出的临时数据库副本上执行，测试结束后自动销毁副本。

## 5.2 功能测试

功能测试主要验证系统各模块能否按照设计要求正常运行。测试按照系统角色划分为普通用户功能测试、商户功能测试与管理员功能测试三部分。测试方法以黑盒测试为主，通过接口请求与数据库状态校验联合确认实际结果。

## 5.3 接口与异常测试

为验证系统在异常情况下的处理能力，进一步对接口进行未授权访问、参数错误、重复审核及越权访问等测试。结果表明，系统能够对非法请求进行有效拦截。

### 5.3.1 未授权访问测试

测试方式：未携带 Token 直接访问受保护接口 `/api/merchant/stalls`。

期望结果：系统返回 401 Unauthorized，拒绝访问请求。

实际结果：接口返回状态码 401，提示信息为“未登录或登录已过期”。

说明：系统能够正确识别未登录请求，并阻止未授权访问。

### 5.3.2 参数错误测试

测试方式：以普通用户身份调用 `POST /api/reviews`，但请求体中缺少 `stall_id` 字段。

期望结果：系统识别请求参数不完整，并返回错误提示。

实际结果：接口返回状态码 400，提示信息为“摊位 ID 格式错误”。

说明：系统能够对无效输入进行校验，避免异常数据直接写入数据库。

### 5.3.3 重复审核测试

测试方式：管理员先审核通过一条摊位新增申请，然后再次对同一条记录执行审核通过操作。

期望结果：系统提示该记录已审核，阻止重复操作。

实际结果：接口返回状态码 400，提示信息为“该提交已审核，无法重复操作”。

说明：系统对审核状态进行了有效限制，能够避免重复审批导致的数据不一致问题。

### 5.3.4 越权访问测试

测试方式：使用普通用户 Token 访问管理员接口 `GET /api/admin/users`。

期望结果：系统拒绝访问并返回权限不足提示。

实际结果：接口返回状态码 403，提示信息为“无权限执行此操作”。

说明：系统的 RBAC 权限控制机制能够有效限制越权访问，管理员接口不会被普通用户调用。

## 5.4 轻量性能与可用性分析

由于本系统属于本科毕业设计项目，未开展大规模并发压测，因此性能评估采用轻量级接口采样与定性分析相结合的方法。需要说明的是，下述时间数据来自本地开发环境中的 Flask `test_client` 测试，主要反映后端接口处理耗时，不包含公网传输与浏览器渲染时间。

### 5.4.1 地图加载相关性能

在当前 120 个摊位点位规模下，系统地图相关数据接口响应较快。热力图接口 `/api/heatmap?mode=density` 连续 20 次采样的平均处理时间为 3.73 ms，95 分位耗时为 4.89 ms，说明后端能够在当前规模下较快完成热力图权重计算与结果返回。结合平台当前数据规模，可认为其能够支撑毕业设计场景下的地图可视化需求。

### 5.4.2 数据查询效率

摊位列表接口 `/api/stalls?page=1&page_size=20` 连续 20 次采样的平均处理时间为 30.74 ms，95 分位耗时为 33.15 ms；管理员摊位列表接口 `/api/admin/stalls?page=1&page_size=20` 的平均处理时间为 10.91 ms。系统采用 SQLite 存储，并结合分页参数进行查询，在 100 条以上业务数据规模下仍能保持较稳定的查询效率，能够满足中小规模夜市平台的基本使用要求。

### 5.4.3 典型接口响应情况

典型接口采样结果如表 5-7 所示。

表 5-7 典型接口轻量性能采样结果

| 接口 | 采样次数 | 平均耗时/ms | 95 分位/ms | 最大耗时/ms |
| --- | ---: | ---: | ---: | ---: |
| `POST /api/auth/login` | 20 | 1.60 | 2.03 | 4.11 |
| `GET /api/stalls?page=1&page_size=20` | 20 | 30.74 | 33.15 | 36.15 |

从测试结果看，登录接口响应最快；摊位列表接口由于涉及分页、评分统计和排序处理，耗时相对更高，但整体仍保持在本地环境可接受范围内。综合来看，系统在当前数据规模与部署方式下具备较好的基础可用性。

## 5.5 结果讨论与改进点

通过上述测试可以得出以下结论。

1. 功能完整性较好。系统已经实现地图展示、摊位管理、评价系统、审核流程、通知机制与审计日志等主要功能，且各模块在测试中均能稳定运行。
2. 权限控制有效。系统通过基于角色的访问控制机制，有效区分了普通用户、商户用户和管理员的操作边界，未出现越权执行管理操作的情况。
3. 业务流程闭环可用。测试验证了两个核心流程能够完整执行：一是“商户提交摊位申请→管理员审核→摊位进入公开列表”；二是“用户提交评价→管理员审核→商户回复→用户收到通知”。
4. 当前性能表现可接受。在 120 个摊位、468 条历史评价的基础数据规模下，典型接口响应较快，能够满足中小规模夜市平台的展示与管理需求。

尽管系统整体运行稳定，但仍存在进一步改进空间。

1. 数据库扩展性有待增强。SQLite 适合当前单机小规模场景，若后续访问量和数据量持续增长，可考虑迁移至 MySQL 或 PostgreSQL。
2. 缓存机制尚未引入。当前地图列表、热力图和统计查询均直接访问数据库，未来可以引入 Redis 等缓存机制提升高频查询效率。
3. 前端性能测试仍可加强。本文的性能分析主要集中在后端接口处理时间，后续可进一步结合浏览器自动化工具，对地图渲染、筛选交互和页面首屏加载时间进行量化测试。
4. 自动化测试体系需要继续完善。目前已实现基于 `test_client` 的接口级验证，但后续仍可补充单元测试、前端端到端测试与持续集成测试流程，提高项目的可维护性与回归测试效率。

## 5.6 本章小结

本章对智慧夜市空间信息可视化平台进行了系统测试与结果分析。测试基于 Windows 10 本地开发环境，在包含 54 个用户、120 个摊位和 468 条评价的基础数据集上开展，并通过临时数据库副本补充了提交审核、评价回复、通知与日志等动态业务数据。测试结果表明，系统 19 项功能与异常用例均通过验证，说明平台能够稳定实现预期功能；同时，RBAC 权限控制机制有效，审核与评价互动流程运行顺畅。结合轻量性能采样结果可知，系统在当前数据规模下具备较好的响应速度与基本可用性，能够满足中小规模夜市平台的信息展示与管理需求。
